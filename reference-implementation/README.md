# Reference Implementations

A framework is only valuable if it can be applied consistently in real-world scenarios.

The Reference Implementations directory demonstrates how the principles, architecture, governance, patterns, templates, and AI workflows defined throughout Documentation Systems Lab can be combined to create a complete, production-quality documentation system.

Unlike isolated examples, each reference implementation represents a cohesive documentation experience built using the standards established elsewhere in this repository.

These implementations are intended to answer an important question:

> **What does a well-designed documentation system actually look like?**

---

# Purpose

Reference implementations exist to demonstrate the practical application of the Documentation Systems Lab framework.

They provide complete examples of how to:

- Organize documentation around user goals.
- Apply a consistent content model.
- Implement documentation patterns.
- Produce canonical API examples.
- Build scalable information architecture.
- Apply governance and editorial standards.
- Support AI-assisted documentation and review.
- Create documentation that is maintainable, discoverable, and reusable.

Every document within a reference implementation should demonstrate—not merely describe—the principles documented elsewhere in this repository.

# Current Reference Implementations

## Atlas Commerce *(In Progress)*

Atlas Commerce is a fictional payment platform created specifically for Documentation Systems Lab.

It exists solely to demonstrate modern API documentation practices using a realistic, end-to-end developer platform.

Atlas Commerce includes examples of:

- Developer onboarding
- Authentication
- Payments
- Refunds
- Customers
- Orders
- Subscriptions
- Webhooks
- Error handling
- OpenAPI specifications
- SDK documentation
- Developer guides
- Reference documentation

Although the APIs, payloads, and workflows are fictional, they are designed to reflect realistic implementation patterns commonly found in modern developer platforms.

# Design Philosophy

Reference implementations are not intended to showcase a fictional product.

They exist to showcase documentation architecture.

Every page should demonstrate one or more concepts described elsewhere in this repository, including:

- Documentation Principles
- Documentation Content Model
- Documentation Style Guide
- Documentation Patterns
- Publication Review Framework
- Documentation Maturity Model

Readers should be able to move seamlessly between the framework documentation and the reference implementation to understand both the reasoning behind a documentation decision and its practical application.

---

# Relationship to the Framework

Documentation Systems Lab is organized into complementary layers.

```text
Documentation Principles
        │
        ▼
Documentation Content Model
        │
        ▼
Documentation Patterns
        │
        ▼
Templates
        │
        ▼
Publication Review Framework
        │
        ▼
```

The framework defines **how** documentation should be designed.

The reference implementation demonstrates **what those design decisions produce**.

---

# What Makes a Reference Implementation Different?

A reference implementation is not simply a collection of sample pages.

It is a complete documentation ecosystem built according to a consistent architectural philosophy.

Each implementation should demonstrate:

- Capability-based navigation
- Consistent page types
- Canonical examples
- Cross-linking between related concepts
- Reusable documentation patterns
- Progressive disclosure
- Comprehensive implementation guidance
- AI-friendly content structure

Every documentation decision should be intentional and traceable back to the architectural and governance documents within this repository.

# Guiding Principle

The purpose of a reference implementation is not to prove that the framework can produce documentation.

It is to demonstrate that the framework can produce **excellent** documentation.

Every page should reinforce the central philosophy of Documentation Systems Lab:

> Documentation is not a collection of pages.

> It is an interconnected knowledge system designed to help people accomplish real work with confidence.
